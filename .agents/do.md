# `do` skill project rules

Picked up by the `/do` skill at `Phase A` and folded into the context bundle as `## Project Rules`. Anything not specified here falls through to the General Strategies in the skill body.

## Toolchain

| Knob | Command |
|---|---|
| Build (assemble all modules) | `./gradlew assemble` |
| Build a single SDK module | `./gradlew :<module>:assemble` (e.g. `:auth:assemble`, `:player:assemble`) |
| Test (full suite) | `./gradlew test --continue` |
| Test a single module | `./gradlew :<module>:test` (e.g. `:auth:test`) |
| Static analysis (detekt) | `./static-analysis/run-detekt.sh` |
| Formatter check (lint mode) | `./static-analysis/run-ktfmt.sh -l` — same command CI uses; never auto-formats |

The CI workflows (`.github/workflows/unit-test.yml`, `.github/workflows/lint.yml`) are the canonical reference if a command in this file ever drifts.

## Pre-commit hook

The repo ships a ktfmt pre-commit hook at `static-analysis/git-hooks/pre-commit`. It must be installed locally before committing — install via `./local-setup.sh` (creates `.git/hooks/pre-commit` and offers to set up `.git-blame-ignore-revs`). The `/do` skill MUST verify `.git/hooks/pre-commit` exists and is executable in Phase B; if missing, stop and tell the user to run `./local-setup.sh`.

Once installed, the hook formats staged Kotlin content on commit. **Never invoke `./static-analysis/run-ktfmt.sh` to rewrite files manually** — the hook owns formatting. Use `./static-analysis/run-ktfmt.sh -l` only as a read-only check.

## Banned commands

None. The SDK is library code with no release-signing requirements (publication happens via GitHub Actions workflows like `create-release.yml`). Both `assemble` and `test` are safe locally.

## Atomic-unit exception

A single function/component that can't be split without breaking its identity may live in one production commit even if it exceeds 100 lines. For this repo specifically:

- Generated code (e.g. tidalapi OpenAPI-generated client surfaces in `tidalapi/src/main/.../generated/`).
- Schema migrations (Room migrations in any module that uses Room).
- `@Composable` functions in demo apps (`*/apps/demo/`) — core SDK modules don't have Compose UI, but the demo apps under each SDK module may.

## Language style routing

All Kotlin (`.kt` / `.kts`) code — production and test — must conform to both:

1. `/review-kotlin` — general Kotlin idioms (null safety, collections, data classes, immutability, coroutines).
2. The repo-local `.agents/checks/*.md` rules — the SDK-adapted Android / coroutine / build patterns.

`/review-kotlin-tidal` (the TIDAL Android client's review skill) is **not** authoritative for the SDK — many of its rules are app-feature-shaped (`feature/upload/`, `Market` design system, `:library` aggregator) and don't apply here. Use the local `.agents/checks/` rules instead; they're already the SDK-adapted distillation.

## Testing conventions

- **Test framework**: JUnit 5 + `mockito-kotlin` + AssertJ-style assertions (or `assertk` where present).
- **Coroutine tests**: `runTest` from `kotlinx-coroutines-test`. Never `runBlocking` in a test body.
- **Main dispatcher swap**: when the code under test touches `Dispatchers.Main`, set up a Main dispatcher swap (`Dispatchers.setMain` / `resetMain` in `@BeforeEach` / `@AfterEach`, or a reusable JUnit 5 extension).
- **Fakes over mocks**: hand-written `FakeXxx` implementations are preferred for SDK-owned interfaces. Reserve `mock<T>()` for Android framework types (`Context`, `Intent`) and final third-party classes (Retrofit `Response`, OkHttp interceptor types).
- **Assert state, not interactions**: prefer state-based assertions over `verify(...)` on interaction patterns.

## Branch naming

- **Linear issue mode**: `michael/<TASK_ID>-<slug>` (slug derived from the issue title).
- **Task description mode**: `michael/<TASK_ID>` (the `task-<short-slug>` generated by the skill).

Slug rules: lowercase, hyphenated, no punctuation, max 50 chars. The base branch is `main` (not `develop`).

## Module paths

SDK is multi-module. Top-level modules:

- `:auth`, `:bom`, `:common`, `:eventproducer`, `:player`, `:template`, `:tidalapi`.
- Each module may have a `apps/demo/` sub-module (e.g. `:auth:apps:demo`).
- Convention plugins live in `buildlogic/` (composite build).

When making a change, the implementer should:

- Scope test runs to the changed module(s) for faster iteration (`./gradlew :auth:test`).
- Run the **full** test suite (`./gradlew test --continue`) at the final verification step before declaring the task done.
- Use `static-analysis/run-detekt.sh` for static analysis — it inspects the whole repo, fast enough to run on every commit.

## Code review per commit

Every commit is reviewed via `sq agents review --codex "HEAD~1...HEAD"`. The `.agents/checks/` rules in this repo are automatically loaded by the runner. Fix any issues, amend, re-run until clean. No separate refactor commits.
